# Transaction이란?

트랜잭션은 은행 업무나 데이터베이스에서 더 이상 쪼개질 수 없는 최소의 작업단위를 뜻하며,
더 자세하게 데이터베이스의 트렌젝션은 데이터베이스의 상태를 변화시키기 위해서 수행하는 작업의 단위를 의미합니다.  
트랜잭션은 원자성, 일관성, 독립성, 지속성 4가지의 특징을 가지고 있습니다.(ACID)

### 원자성, Atomicity
한 트랜잭션의 연산들이 모두 성공하거나, 모두 실패하는 것을 보장하는 것을 의미합니다.  
(부분적인 성공, 실패가 없다라는 의미)
### 일관성, Consistency
데이터는 미리 정의된 규칙에서만 조작이 가능하도록 보장하는 것을 의미합니다.
### 독립성, Isolation
트랜잭션을 수행 시 다른 트랜잭션의 연산 작업이 끼어들지 못하도록 보장하는 것을 의미합니다.
### 지속성, Durability
성공적으로 수행된 트랜잭션은 영원히 반영되어야 함을 의미합니다.

### 트랜잭션의 격리 수준(Isolation Level)이란?
트랜잭션의 격리 수준이란 동시에 여러 트랜잭션이 수행될 때,
특정 트랜잭션이 다른 트랜잭션에서 변경하거나 조회하는 데이터를 볼 수 있도록 허용할지 말자를 결정하는 것입니다.  
격리 수준은 다음과 같이 4가지로 정의 할 수 있습니다.
- READ UNCOMMITTED(커밋되지 않은 읽기)
- READ COMMITTED(커밋된 읽기)
- REPEATABLE READ(반복 가능한 읽기)
- SERIALIZABLE(직렬화 가능)

순서대로 READ UNCOMMITTED의 격리 수준이 가장 낮고 SERIALIZABLE의 격리 수준이 가장 높습니다.
트랜잭션의 격리 수준이 낮을수록 더 많은 문제가 발생합니다.

# SQL Injection이란?

SQL 인젝션(또는 SQL 삽입, SQL 주입)은 코드 인젝션의 한 기법으로 클라이언트의 입력값을 조작하여
서버의 데이터베이스를 공격할 수 있는 공격방식을 말합니다. 주로 사용자가 입력한 데이터를 제대로 필터링,
이스케이핑하지 못했을 경우에 발생합니다.  
쉬운 공격 난이도에 비해 발생하는 피해는 어머어마하기 때문에
시큐어 코딩을 하는 개발자라면 가장 먼저 배우게 되는 내용입니다.  
이러한 인젝션 계열의 취약점들은 테스트를 통한 발견은 어렵지만 스캐닝툴이나 코드 검증절차를 거치면 보통 쉽게 발견되기 때문에 탐지하기는 쉬운 편이라고 합니다.

### 공격 목적
SQL 인젝션의 대표적인 목적으로는 다음과 같은 것들이 있습니다.
1. 정보 유출
2. 저장된 데이터 유출 및 조작
3. 원격 코드 실행
4. 인증 우회

### 공격 방법
SQL 인젝션 공격의 대표적인 방법으로는 로그인 폼을 대상으로 하는 공격이 있습니다.
이를 통해 정상적인 계정 정보 없이도 로그인을 우회하여 인증을 획득할 수 있습니다.  

또한 시스템에서 발생하는 에러 메세지를 이용해 공격하는 방법도 존재합니다. 이외에도 다양한 SQL 인젝션 공격 방법이 존재합니다.

### 대응 방안
SQL 인젝션 공격에 대한 대응 방안으로는 대표적으로 3가지를 꼽을 수 있습니다.
1. 입력 값에 대한 검증
2. Error Message 노출 금지
3. Prepared Statement 구문사용
위와 같은 방법을 통해 어느정도 SQL 인젝션 공격에 대한 공격이 가능합니다.

# 동적쿼리란 무엇이고 언제 사용하는지 설명해주세요.

동적쿼리란? 동적쿼리란 실행시에 쿼리문장이 만들어져 실행되는 쿼리문을 말합니다. 쿼리문이 변하냐 변하지 않느냐에 따라 변하지 않으면 정적쿼리 변한다면 동적쿼리로 생각하시면 됩니다.
대표적으로 QureyDSL이 있으며  

    @Override
    public List<Employee> findAllBySearch(String search, String category) {
        BooleanBuilder builder = new BooleanBuilder();
        builder = switch (category) {
            case "직원번호" -> builder.and(employee.employeeNumber.contains(search));
            case "이름" -> builder.and(employee.name.contains(search));
            case "직급" -> builder.and(employee.position.contains(search));
            case "전화번호" -> builder.and(employee.phoneNumber.contains(search));
            case "이메일" -> builder.and(employee.email.contains(search));
            default -> throw new CustomException(WRONG_CATEGORY_ERROR);
        };
        return jpaQueryFactory
                .selectFrom(employee)
                .where(
                        builder
                )
                .orderBy(employee.name.asc())
                .fetch();
    }  
이러한 쿼리문을 말합니다.  
쿼리문의 정렬 값이나 조건 값 등이 상황에 따라 유동적으로 변해야될 필요가 있을 때 사용합니다.

# Redis의 장단점에 대해 설명해주세요.
### Redis란 무엇인가?

Key, Value 구조의 비정형 데이터를 저장하고 관리하기 위한 오픈 소스 기반의 비관계형 데이터 베이스 관리 시스템 (DBMS)

레디스는 인메모리 방식의 데이터베이스로서 빠른 읽기/쓰기 속도를 보장한다.
인메모리 방식을 사용한다고 해서 서버가 다운되면 모든 데이터가 날라간다고 생각하기 쉬우나
기본적으로 스냅샷 기능을 제공하여 파일로 저장되므로 저장된 시점으로 복구가 가능하다.

또한 다양한 자료구조를 지원하며 싱글 스레드 방식으로 작동하여 race condition 문제로부터 자유롭다.  
하지만 이 방식은 단점이 될 수도 있는데 시간이 오래 걸리는 요청이 온다면 뒤에 오는 요청들은 쌓이게 된다.

# 정규화란 무엇인가?  

정규화는 이상현상이 있는 릴레이션을 분해하여 이상현상을 없애는 과정이다.  
이상현상이 존재하는 릴레이션을 분해하여 여러개의 릴레이션을 생성하게 되는데, 이를 단계별로 구분하여 정규형이 높아질 수록 이상현상은 줄어들게 된다.  
  
### 장점
* 데이터베이스 변경 시 이상현상을 제거할 수 있다.
* 정규화된 데이터베이스 구조에서는 새로운 데이터 형의 추가로 인한 확장시, 그 구조를 변경하지 않아도 되거나 일부만 변경해도 된다.
* 데이터베이스와 연동된 응용 프로그램에 최소한의 영향만을 미치게 되어 응용프로그램의 생명을 연장시킨다.<br>  

### 단점  
* 릴레이션의 분해로 인해 릴레이션 간의 JOIN연산이 많아진다.  
* 질의에 대한 응답 시간이 느려질 수도 있다. 데이터의 중복 속성을 제거하고 결정자에 의해 동일한 의미의 일반 속성이 하나의 테이블로 집약되므로 한 테이블의 데이터 용량이 최소화되는 효과가 있다.   
* 데이터를 처리할 때 속도가 빨라질 수도 있고 느려질 수도 있다.  
* 조인이 많이 발생하여 성능 저하가 나타나면 비정규화를 적용해야할 수도 있다.<br/>

## 정규화 단계

데이터베이스 정규화는  1NF, 2NF, 3NF, BCNF, 4NF, 5NF 로 구분됨

### 단계별 특징

**제 1 정규화(1NF)**<br/>
- 모든 열이 **원자값**을 가지도록 테이블을 구성
- 테이블에서 중복된 데이터를 제거하여 테이블의 모든 열이 한 하기 값만 포함하도록 수행

**제 2 정규화(2NF)**<br/>
- 모든 열이 **완전 함수적 종속을 만족(부분 함수 종속 제거)** 하도록 테이블을 구성
- 테이블의 기본 키를 제외한 모든 열이 기본 키에 대해 완전히 종속적이여야 한다
- 하나의 테이블에 두 개 이상의 기본 키가 있으면 이를 분리하여 각 테이블에서 하나의 기본 키만 사용

**제 3 정규화(3NF)**<br/>
- 테이블의 모든 열이 **이행적 함수적 종속을 제거**
- 테이블의 열이 기본 키에 대해서만 종속되어야 한다

**보이스-코드 정규형(BCNF)**<br/>
- 테이블의 모든 **결정자**가 후보 키이거나 기본 키인 것을 의미
- 이는 테이블에서 한 열의 값을 변경할 때 다른 열의 값을 변경하지 않도록 하는 것
- 
**제 4 정규화 (4NF)**<br/>
- 테이블의 모든 **다중값 종속**을 처리
- 테이블에서 어떤 열의 값을 변경하면 다른 열의 값이 변경될 가능성이 있는 **다중값 종속**이 있는 경우 해당 종속을 분리하여 새로운 테이블 제작

**제 5 정규화(5NF)**<br/>
- 테이블에서 **조인**을 통해 얻을 수 있는 데이터를 모두 포함하는 다중 집합 종속이 있는 경우
- 다중값 종속성을 분리하여 새로운 테이블로 만들어 정규화를 수행
  

# 데이터의 무결성이란?   

데이터 무결성은 데이터의 정확성, 일관성, 유효성이 유지되는 것을 의미한다.  
여기서 정확성이란 중복이나 누락이 없는 상태를 뜻하고, 일관성은 원인과 결과의 의미가 연속적으로 보장되어 변하지 않는 상태를 뜻한다.  

만약 데이터베이스에서 데이터 무결성 설계를 하지 않는다면 테이블에 중복된 데이터 존재, 부모와 자식 데이터 간의 논리적 관계 깨짐, 잦은 에러와 재개발 비용 발생 등과 같은 문제가 발생한다.  
그렇기 때문에 DBMS에서 데이터의 무결성이 유지되는 것은 중요한 사항이며, 주로 데이터에 적용되는 연산에 제한을 두어 데이터의 무결성을 유지한다.

# View와 Table의 차이점

## Table

- 데이터를 저장하는 데 사용되는 데이터베이스 객체
- 일반적으로 행과 열의 형태로 데이터를 저장
- 테이블은 특정 유형의 데이터를 저장하는 데 사용될 수 있으며, 특정 업무나 응용프로그램에서 필요한 정보를 저장하고 검색하는 데 사용됩니다.

## View

- 하나 이상의 테이블의 데이터를 기반으로 생성되는 가상 테이블
- 실제 데이터를 저장하지 않지만, 쿼리문을 사용하여 여러 테이블에서 필요한 데이터를 조합하여 가져올 수 있음
- 저장된 데이터를 변경하지 않기 때문에, 데이터 일관성을 유지하고, 중복을 제거하며, 쿼리문의 복잡성을 감소시키는 등의 이점을 가지고 있음

### View를 사용하는 이유

**데이터 보안**

- 특정 사용자에게 데이터의 일부분만 제공할 수 있음
- 민감한 정보를 포함하는 열을 숨기거나, 특정 행만 보여줄 수 있음

**데이터 무결성 유지**

- 기존의 데이터를 변경하지 않기 때문에, 데이터 무결성이 유지

**쿼리문의 단순화**

- 여러 테이블을 조인해야 하는 복잡한 쿼리문을 단순화 가능

**재사용성**

- 여러 쿼리문에서 동일한 데이터를 사용할 수 있기 때문에, 코드의 재사용성이 향상

### View의 한계점

- View를 사용하면, 실제 데이터가 아닌 가상 데이터를 사용하기 때문 → 데이터를 가져오는 속도가 느려질 수 있음
- View가 사용하는 쿼리문이 복잡해지면, 쿼리문의 실행 속도도 느려질 수 있습니다

# 객체 관계 매핑(ORM)이란?
### Object Relational Mapping, ORM
객체와 관계형 데이터베이스의 데이터를 자동으로 매핑(연결)해주는 것을 말합니다.  
객체 모델과 관계형 데이터베이스 모델 간에는 불일치가 존재합니다.  
ORM을 이용하지 않으면 개발자는 직접 이 불일치 문제를 해결해주어야 합니다.  
개발자는 ORM을 통해 객체 간의 관계를 바탕으로 SQL을 자동으로 생성하여 불일치를 해결해줍니다.
개발자는 ORM을 이용해 객체를 통해 간접적으로 데이터베이스를 다룰수 있습니다.

### 장점
- 객체 지향적인 코드를 작성할 수 있고, 이로 인해 더 직관적이게 되며 비즈니스 로직에 집중할 수 있게 됩니다.
- 재사용 및 유지보수 등의 편리성이 증가합니다.
- DBMS에 대한 종속성이 줄어듭니다.

### 단점
- 완벽한 ORM으로만 서비스를 구현하기 힘듭니다.
- 프로시저가 많은 시스템에선 ORM의 객체 지향적인 장점을 활용하기 어렵습니다.

# 데이터베이스의 조회성능을 개선하는 방법

### 인덱스 추가

- 테이블에 인덱스 알고리즘 적용시 특정 컬럼을 검색할 때 더 빠르게 조회할 수 있음
- **인덱스**는 데이터베이스 엔진이 특정 컬럼의 값을 저장하는 방법

### 쿼리 최적화

- 사용중인 쿼리를 보다 효율적으로 최적화하여 빠르게 조회할 수 있도록 한다
- 쿼리 최적화는 쿼리 실행 계획을 분석하여 최적화하는 것으로, 인덱스를 활용하는 등의 방법을 사용

### 캐싱

- 데이터베이스에서 조회한 결과를 캐시에 저장하여 조회
- 캐시는 메모리에 데이터를 저장하는 것
- 조회할 때 먼저 캐시에서 데이터를 찾아보고, 없을 경우 데이터베이스에서 조회

### 파티셔닝

- 데이터베이스 테이블을 파티션으로 나누어 사용
- 테이블의 데이터를 논리적으로 나누어 저장하는 것
- 특정 조건에 따라 나누거나 일정한 크기로 나누는 등의 방법을 사용

### 스케일 아웃

- 데이터베이스 서버를 여러 대로 분산
- 여러 대의 서버에 데이터베이스를 분산시켜 처리하는 것
- 부하 분산이나 장애 대응 등의 목적으로 사용

### 스케일 업

- 단일 서버에서 사용하는 하드웨어 자원을 업그레이드 → 데이터베이스 처리 능력을 향상

# RDB vs No-SQL

## RDB

- 테이블 형식으로 데이터를 저장
- 각 테이블은 고유한 기본 키를 가짐
- SQL을 사용하여 데이터를 저장, 검색 및 관리
- ACID 원칙을 따름
- 고성능, 안정성 및 확장성이 뛰어남

## NoSQL 데이터베이스

- SQL을 사용하지 않음
- 스키마 없이 데이터를 저장
- 대규모 분산 데이터를 처리하기 위한 목적으로 설계
- 다양한 데이터 모델 (키-값, 문서, 그래프, 열 기반)을 제공
- 빠른 데이터 검색 및 저장을 위한 분산 아키텍처를 제공
- ACID 원칙을 모두 따르지 않을 수 있음
- 일관성과 가용성이 더 중요

### 각각 장단점

### RDB

- **안정적**이며 **일관성**이 뛰어나며, **데이터의 정확성을 보장**
- 고정된 데이터 스키마가 있어 **유연성이 떨어질 수 있음**
- 대규모 데이터 처리에 한계가 있음

### NoSQL

- 대규모 데이터를 **빠르게 처리**하며 **유연성이 높음**
- **일관성**이 떨어지고, **데이터 정확성에 대한 보장이 낮을 수 있음**

### 언제 사용하나?

### RDB

대부분의 경우 데이터의 일관성과 정확성이 중요한 경우

### NoSQL

대규모의 비정형 데이터를 처리하고 빠른 검색 및 저장을 필요로 할 때 적합

## 하이브리드?

최근에는 RDB와 NoSQL의 기능을 결합한 NewSQL이라는 새로운 유형의 데이터베이스가 등장하여 둘의 장점을 모두 활용할 수 있는 장점을 갖고 있습니다.
